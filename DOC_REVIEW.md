# 设计文档审查说明（需求 + 技术方案）

本文件用于 **feature 分支** 上 `dev-doc/docs` 变更时的自动审查，优先级高于默认规则与 `CLAUDE.md`。
业务仓库可复制后按实际业务（支付、账务、风控、开户等）增删条款。

## Agent 审查流程（控制 Actions 轮数）

自动化审查时 **只审** `.github/claude-doc-diff.patch` 所列变更，以及 meta 中列出的文档正文；不要扫全库。

1. 读 `DOC_REVIEW.md` 一次，按下方 Important 清单对照文档  
2. 发 **一条** 总结（有 PR 则 `gh pr comment`，否则 commit comment），首行 `Important: N，Nit: M`  
3. 若开启 inline（`post_inline_comments=true` 且有关联 PR）：对最多 **8** 条 Important 在文档对应行发 PR review comment（Files 视图可见）  
4. Nit ≤ 5 条；无 inline 时 Important 尽量在总结中带文档路径与章节/行号  

## Important（必须修复）

以下任一情况必须标为 **Important**：

### 需求（PRD）完备性

- 缺少背景/目标、范围或明确的非目标，导致实现边界不清
- 关键业务流程缺少角色、触发条件、成功/失败路径或状态说明
- 核心业务规则（金额、费率、限额、币种、权限、幂等）表述模糊或自相矛盾
- 无验收口径 / 验收标准无法检验
- 涉及资金变动却未说明冻结/扣款/解冻/回滚或补偿预期

### 技术方案完备性

- 模块/服务边界不清，与现有模块职责冲突或未说明依赖
- 缺少关键数据模型、状态机，或状态迁移不闭合（存在无出口/不可达状态）
- 写路径（下单、调账、回调、定时任务）未覆盖鉴权、幂等、审计中的必要项
- 异常、重试、补偿、对账口径缺失，可能导致资金或状态不一致
- 对外接口/事件缺少必要字段约定或错误码语义

### 需求与设计一致性

- 技术方案未覆盖 PRD 中的范围项，或实现了明确非目标内容却未解释
- PRD 与设计对同一规则（费率、状态、角色权限、资金语义）描述不一致
- 关键约束（合规、安全、多租户）仅在一侧出现

### 金融与安全表述

- 文档暗示可用浮点做金额运算、或余额读-改-写无并发防护
- 回调/内部接口对公网暴露却无签名/白名单/鉴权设计
- 敏感字段日志/接口回显策略缺失或错误

## Nit（建议完善，不单独视为阻塞）

- 章节结构、命名、图表缺失但上下文仍可理解
- 非关键路径的性能/体验描述不足
- 与 `CLAUDE.md` 或仓库文档风格不一致
- 可补充的示例、链路图、对照表

每条审查最多报告 **5** 条 Nit；更多则在总结中写「另有 N 条类似建议」。

## 不报告（勿浪费轮次）

- 纯格式/错别字（除非导致语义错误）
- 与本次 diff 无关的历史文档问题
- 实现层代码问题（应留给代码 PR Review）；本文档审查只评需求与方案本身

## 每次必查

1. 变更文件中是否同时或配套更新了需求与设计；若只改一侧，是否说明原因  
2. 资金相关能力：冻结/扣款/解冻、幂等键、审计字段是否写清  
3. 范围/非目标是否仍然成立，有无悄悄扩大范围  
4. 失败与补偿路径是否与成功路径同等完整
