# 项目规范（金融）

业务仓库可复制或扩展。Claude Code 各类任务（含 PR Review）会参考本文件。
**仅针对审查的硬性门槛与 Important 定义，以根目录 `REVIEW.md` 为准**（审查时 `REVIEW.md` 优先级更高）。

## 业务与工程原则

- 资金相关逻辑：正确性优先于性能；变更需可测试、可回滚、可追溯
- 禁止为通过构建而删除测试、跳过校验或注释掉风控/鉴权代码
- 生产配置与代码分离；禁止将调试开关、跳过验签等默认值提交到仓库

## 代码质量

- 避免空指针、资源泄漏、吞异常；对外 API 错误信息不得泄露内部实现与敏感数据
- 避免大段注释掉的死代码
- 公共方法需有清晰的失败语义（失败时状态可恢复或进入明确补偿流程）

## 安全与隐私

- 禁止硬编码密钥、证书、数据库密码、第三方 API Key
- 所有外部输入需校验（长度、格式、范围、枚举）；防 SQL 注入、路径遍历、SSRF
- 日志、监控、异常栈：不得输出密码、token、完整证件号、完整卡号、完整手机号
- 接口响应遵循最小必要原则；敏感字段脱敏展示

## 资金与数据（Java 常见）

- 金额使用 `BigDecimal`（或项目统一金额类型），明确 scale 与舍入模式；禁止 `double` 做金额运算
- 账务、余额、订单状态变更需在事务内完成；跨服务需幂等键 + 对账/补偿设计
- 多租户场景：数据访问必须带租户维度，禁止仅靠可选参数过滤
- 关注并发、分布式锁、消息重复消费对余额的影响

## 测试

- 资金、鉴权、状态机变更需有单元或集成测试；边界与失败路径不可省略
- 不删除或弱化既有安全/账务相关测试以通过 CI

## Java / 构建

- 关注事务边界、`@Transactional` 传播行为、异步与事务失效
- Maven/Gradle 构建需通过；不随意禁用安全扫描或静态检查（除非 PR 说明原因）
