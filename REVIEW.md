# 代码审查说明（金融项目）

本文件仅用于 PR Code Review，优先级高于默认审查规则与 `CLAUDE.md`。
业务仓库可复制后按实际系统（支付、账务、风控、开户等）增删条款。

## Agent 审查流程（控制 Actions 轮数）

自动化审查时 **只审 `.github/claude-pr-diff.patch` 中的变更**，不要扫全库。

1. 读 `REVIEW.md` 一次，按下方 Important 清单对照 diff  
2. 发 **一条** PR 总结评论，首行 `Important: N，Nit: M`  
3. 默认 **不发** inline comment（由 workflow `post_inline_comments` 控制）  
4. Nit ≤ 5 条；Important 每条需 `路径:行号`  

轮数不够时：缩小 PR、提高 `max_turns`，或关闭 inline comment。

## Important（必须修复）

以下任一情况必须标为 **Important**，并在 PR 总结中列出（含 `路径:行号`）：

### 资金与账务正确性

- 金额使用浮点类型（`float`/`double`）做金额运算或持久化
- 余额、冻结、扣款、退款缺少事务边界或可能出现重复入账/重复扣款（无幂等键、无唯一约束、无乐观锁/分布式锁）
- 借贷不平、舍入规则不一致、币种/精度处理错误
- 并发下读-改-写余额存在 TOCTOU，且无防护

### 鉴权与访问控制

- 接口/任务未校验调用方身份，或仅依赖前端传入的用户 ID/账户号
- 水平越权：可访问非本人/非本租户的数据（缺 tenantId、orgId、userId 与资源归属校验）
- 垂直越权：普通角色可执行管理、调账、冲正等高危操作
- 内部接口、回调、定时任务对公网暴露且缺少签名/白名单/mTLS

### 敏感数据与密钥

- 代码或配置中出现明文密钥、API Key、私钥、数据库密码
- 日志、异常信息、接口响应中包含：完整卡号、CVV、密码、身份证号、手机号、完整银行卡号、会话 token
- 敏感字段未脱敏展示（应掩码或哈希，且不可逆场景禁止日志打印原文）

### 注入与外部调用

- SQL/MyBatis 使用字符串拼接 IN/WHERE，或 `${}` 拼接用户输入
- 命令执行、SpEL、OGNL、动态脚本执行依赖外部输入
- 对外 HTTP/RPC 调用 URL 或重定向目标可被用户控制（SSRF 风险）
- 反序列化不可信数据（如 Java 原生序列化、不安全 JSON 多态）

### 审计与可追溯

- 资金变动、状态变更、审批操作无审计日志（操作人、时间、前后状态、业务单号）
- 审计日志可被业务代码静默关闭或绕过

### 合规相关实现缺陷

- 关键金融操作无失败回滚或补偿机制，且可能导致资金状态不一致
- 绕过已有关风控/限额/黑名单校验的代码路径

## Nit（建议修复，不单独挡合并）

- 命名、格式、少量重复代码、注释缺失
- 非关键路径的性能微优化
- 测试覆盖不足但**不改变**资金/安全语义
- 与 `CLAUDE.md` 风格规范不一致但不影响正确性的项

每条审查最多报告 **5** 条 Nit；更多则在总结中写「另有 N 条类似建议」。

## 不报告（勿浪费评论）

- 已由 CI 覆盖：编译、单测、Checkstyle/SpotBugs、依赖漏洞扫描（除非 PR **新增**了绕过扫描的配置）
- 纯生成代码目录、`*.lock`、vendor、前端打包产物
- 仅文档/注释/配置文案、无逻辑变更的 PR（除非涉及安全或合规表述错误）

## 每次必查（金融通用）

- 新建或修改的 **写接口**（下单、支付、转账、调账、开户、绑卡）：是否有鉴权 + 幂等 + 审计
- **读接口**是否泄露他人数据；列表/详情是否强制带租户/用户过滤
- 数据库变更：是否有不可逆迁移；大表变更是否考虑锁表与回滚
- 消息队列/回调：是否可能重复消费导致重复动账；重试是否有上限与死信
- 配置变更：是否将生产密钥、调试开关、跳过校验开关带入默认值

## 严重程度

- 发现 **Important** → 在 PR 评论中明确列出，合并前须处理
- 仅 **Nit** → 总结中标注，不阻塞合并
- 不确定但风险高 → 标 Important 并说明依据（文件:行 + 可能后果）

## 审查输出

- 总结首行给出：`Important: N，Nit: M`；无 Important 时写「无阻塞项」
- 资金/安全问题必须引用 `路径:行号`，避免空泛批评
